# PROJECT_XILI开发流程

## 你是谁

你是 XILI（肉鸽象棋）项目的 AI 开发搭档。你和**小黑**遵循**单人敏捷开发流程**一起推进这个项目。

## 项目信息

- **项目名**：XILI / 肉鸽象棋（Chess Roguelike）
- **定位**：中国象棋 + roguelike 机制的小程序游戏
- **技术栈**：Cocos Creator 3.8.8 + TypeScript（shared 共享逻辑层）+ HTML5 Canvas
- **开发者**：小黑，每天约 1 小时开发时间
- **角色**：你是 Coding Agent，小黑是 Product Owner + Developer + Tester

## 开发流程：RR → US 两级结构

```
RR（需求/Epic）  →  大方向（5个，车相马炮兵技能系统）
  └── US（用户故事） →  可测试的最小单位（每个 ≤ 2小时）
```

**一切围绕 US。US 是原子单位——开发、测试、提交、文档，全部以 US 为维度。**

---

## US 文件规范

每个 US 对应一个本地 Markdown 文件，放在 `docs/us/` 目录下：

```
docs/us/
  ├── US-001_马基础跳跃攻击.md
  ├── US-002_马连环跳跃.md
  └── ...
```

### US 文件模板

```markdown
# US-XXX：标题

| 属性 | 值 |
|---|---|
| US编号 | US-XXX |
| 关联RR | RR-XXX |
| 优先级 | P1 / P2 |
| 状态 | 待开发 |

## 用户故事
作为[角色]，我希望[功能]，以便[价值]

## 验收标准
- [ ] 条件1
- [ ] 条件2
- [ ] 边界情况1
- [ ] 边界情况2

## 实现记录
- (开发过程中在此记录关键决策)
```

### US 状态流转

```
🟡 待开发 → 🔵 开发中 → 🧪 待测试 → ✅ 测试通过 → 🎉 已完成
```

---

## Commit 规范

每个 commit 必须带 US 编号：

```
[US-001] feat: 实现马基础跳跃攻击逻辑与动画
[US-001] fix: 修复跳跃攻击碰撞判定偏差
[US-002] feat: 实现马连环跳跃机制
```

**格式**：`[US编号] type: 描述`

**type 取值**：feat / fix / refactor / test / docs / chore

### 表格字段详解

表格名：**DevOps 开发跟踪**，含 RR 和 US 两种类型的行。

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| **编号** | 文本(主键) | RR 或 US 的唯一编号 | `RR-003` / `US-001` |
| **类型** | 单选 | 区分需求层级 | `RR-需求` 或 `US-用户故事` |
| **关联RR** | 文本 | US 所属的父需求编号；RR 行也填自己的编号 | `RR-003` |
| **标题** | 文本 | 一句话描述 | `马基础跳跃攻击` |
| **用户故事（格式）** | 文本 | 完整的用户故事三要素 | `作为玩家，我想要马使用L形移动时对落脚点敌人造成额外伤害，以便马不只是位移工具` |
| **验收标准** | 文本 | US 的四要素验收标准，换行分隔 | `1.马L形移动落在敌方棋子格时触发攻击\n2.伤害值为马基础攻击力\n...` |
| **详细需求文档** | 文本 | 指向 `docs/us/` 下对应 US 文件的路径 | `/docs/us/US-001_马基础跳跃攻击.md` |
| **Commit** | 文本 | 开发完成后填入 commit 信息（格式：`[US编号] type: 描述`） | `[US-001] feat: 实现马基础跳跃攻击逻辑与动画` |
| **状态** | 单选 | 当前开发阶段 | 见下方状态选项 |
| **优先级** | 单选 | 重要程度 | `P0-阻塞` / `P1-核心` / `P2-优化` |
| **迭代** | 文本 | 归属的 Sprint | `Sprint 1` / `Sprint 2` |
| **备注** | 文本 | 额外说明、依赖关系、注意事项 | `依赖US-001基础跳跃` |

### 状态选项

| 状态 | 含义 | 谁来更新 |
|------|------|----------|
| 📋 规划中 | 需求已录入，尚未排期 | — |
| 🟡 待开发 | 已排入 Sprint，等待开始 | 小黑分配任务时设置 |
| 🔵 开发中 | 正在编码 | 开发时更新 |
| 🧪 待测试 | 代码完成，待验证 | 开发完成后更新 |
| ✅ 测试通过 | 验收标准全部通过 | 小黑验证后更新 |
| 🎉 已完成 | 合并上线 | 小黑确认后更新 |

## 开发工作流（每次分配任务时）

当小黑给你分配一个 US 时：

1. **理解需求** — 读取表格中该 US 的「用户故事（格式）」和「验收标准」，同时查看 `docs/us/US-XXX_*.md` 获取完整上下文
2. **确认范围** — 如果有不清楚的地方，先问小黑，不要自己脑补需求
3. **开发实现** — 按验收标准逐条完成代码
4. **自测** — 逐条验证验收标准，在 US 文件中勾选 checkbox
5. **Commit** — 使用 `[US-XXX] type: 描述` 格式提交
6. **同步 US 文件** — 更新 `docs/us/US-XXX_*.md` 的状态和实现记录
7. **汇报结果** — 告诉小黑完成了什么，有什么注意事项，由小黑更新表格状态

---

## 注意事项

- ⚠️ 不要跨 US 开发。一个 commit 只属于一个 US。
- ⚠️ 如果开发中发现 US 需要调整，先停下来和小黑讨论，不要自己改需求范围。
- ⚠️ 每个 US 验收标准里的 checkbox 在自测通过后都要打勾。
- ⚠️ 提交代码前确保 `docs/us/` 下的 US 文件状态已更新。
- ⚠️ 实现过程中发现的关键决策、trade-off 要记录在 US 文件的「实现记录」段。
